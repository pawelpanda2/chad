# Story 57 — Plan

No interactive plan approval for this Story — the user explicitly asked not to be asked and went to sleep, granting full autonomy including "even random" decisions where genuinely unsure. This file records the plan as executed, written by the AI, not reviewed by the user beforehand.

## What already existed to build on

An earlier, separate session (not this Story) built out a full TypeScript rewrite of the .NET Content Provider: `packages/content-provider/{core,net-adapter,files,mongo,entry,api}` and a standalone `packages/cp-gui` component library. `cp-gui`'s components (`TextView`, `FolderView`, `ContentProviderBrowser`) are close in spirit to what's needed here but were built for a different context — a standalone desktop-replacement tool with a free repo picker and `cp-plugin` (local file/folder/terminal opening) integration. Neither of those fit a shared, multi-user web dashboard, so this Story writes new, dashboard-native components rather than importing `cp-gui` directly, while reusing `cp-core`'s types and `cp-entry`'s routing (the actual data-access layer).

## Scope decisions (made autonomously, per the user's explicit permission)

1. **Repo is locked to the current authenticated user's own `repoGuid`, not a free picker.** The dashboard has an established per-user Content Provider data isolation model (`documentation/dashboard/common/features/chad-user-data-isolation.md`) — every other Content-Provider-touching endpoint scopes strictly to `getCurrentUserFromCookies()`'s `repoGuid`. Blazor's `Repos.razor` has a combobox over ALL repos because it's a single-operator desktop tool; exposing that in a shared web dashboard would leak one user's private repo to another logged-in user. The "repo" toolbar element the user asked to keep is rendered as a read-only label (the repo's own logical name, e.g. "chad_pawel_f"), not a switchable control.
2. **Per-item toolbar buttons (Folder/Content/Config/Terminal, GoogleDoc/Tts, "Add" forms) are omitted entirely**, per the user's own words ("i pod nimi sa przyciski ktore sa tutaj niepotrzebne" — the buttons below are unnecessary here). These all either talk to `cp-plugin` (a local desktop helper with no meaning inside a web dashboard) or are write operations (`Put`/`PostParentItem` — not wired into `cp-api` yet, out of scope).
3. **Back/Forward are real browser-style history over already-fetched items** (array + index pointer, navigating to a new item after going back truncates forward history) — not Blazor's own broken triple-button (all three of its buttons call the same handler; not worth replicating a confirmed dead-code bug a second time in a second codebase).
4. **The old `/dashboard/folders` page and its `/api/folders` route are fully replaced**, not added alongside. The existing page was a narrow, hardcoded actions/leads debug viewer — not a generic browser — and the user's request ("umieść w zakładce Folders GUI content providera") describes replacing that tab's content wholesale. `getCurrentUserForms` in `cp-flow.ts` is left in place (unused after this change, but a shared module function, not deleted speculatively without checking every possible future caller).
5. **`cp-entry` (and transitively `cp-core`, `cp-net-adapter`) added as a new dashboard dependency**, wired into Next.js the same way `dba` already is (`transpilePackages` + turbopack `resolveAlias`). Server-side only (a new API route) — never imported into a client component, matching the project's existing architecture rule that dashboard/cp-gui code only ever talks to Content Provider through a server-side layer.
6. **No new backend/routing decisions**: `cp-entry`'s default backend routing (repo → `net-adapter`, i.e. the real .NET API) is untouched — this Story only adds a new consumer of the existing, already-verified `cp-entry` → `cp-net-adapter` → real .NET path, scoped per-request to the authenticated user's `repoGuid` (the same safe explicit-parameter pattern `cp-flow.ts` already uses — no `runWithRepoContext` needed since `cp-entry`'s methods already take `repoGuid` explicitly).

## Correction (Input 2) — dropped `cp-entry`/`cp-core`, reverted to `cp-flow.ts`

Point 5 above was wrong in practice: `cp-entry`/`cp-core` were added as a new `packages/dashboard` dependency, but the dashboard's **Docker** build (`packages/dashboard/Dockerfile`) never builds those packages' `dist/` output inside the container before running `next build` — only `dba` gets that treatment. First deploy attempt failed: `Module not found: Can't resolve 'cp-entry'` / `'cp-core'`. The user's actual instruction, once this surfaced, was more fundamental than "fix the Dockerfile": **don't add `cp-entry`/`cp-core` as a dependency at all** — "to guid ma dzialac wciaz ze starym net-content-provider" (this GUI should keep working against the OLD net-content-provider, i.e. the existing, already-deployed .NET API, through the existing integration path, not a new one).

Reverted: `packages/dashboard/package.json` (removed `cp-core`/`cp-entry`), `packages/dashboard/next.config.ts` (removed their `transpilePackages`/turbopack `resolveAlias` entries), and the Dockerfile build-step addition that had been drafted to fix the original failure (no longer needed once the dependency itself is gone).

Replaced with: a new `getItemByLoca(repoGuid, loca)` function added directly to `packages/dashboard/app/api/flow/cp-flow.ts` — the file's own header comment already states it's "the ONLY place where Content Provider args are manually constructed." It calls the file's existing low-level `invokeCp()` helper (same `fetch` to `${CONTENT_PROVIDER_API_URL}/invoke`, same mechanism every other dashboard endpoint already uses in production) directly, bypassing the higher-level `invokeCpWithTrace()` (which only extracts a `.result` string, built for the Body-only forms/leads flows and doesn't match `GetItem`'s real `{Body, Settings}` response shape). `/api/folders/route.ts` now calls `getItemByLoca` instead of `cp-entry`'s `entry.GetItem`. No new workspace dependency, no Dockerfile change, no build-order concern — the Folders tab now uses exactly the same underlying wiring as every other existing dashboard page.

This does NOT touch or invalidate the earlier session's `cp-entry`/`cp-files`/`cp-mongo`/`cp-api` TypeScript rewrite work — that all still exists in the repo, built and verified on its own, just not consumed by the dashboard.

## Correction 3 (Input 5) — nested frame, real bugs from actual use, Dev Panel investigated

The user actually used the deployed page this time and reported four things:

1. **"wstaw go jeszcze w dodatkowa ramke"** — the Repo/Loca/nav block, previously separated from the rest of the content by just a `border-b`, is now wrapped in its own nested bordered `rounded-lg border` frame.
2. **"wszystko kreci sie w nieskonczonosc, dopiero kliknięcie GO odblokowało"** — root cause found by reading the render logic, not guessed: `{!currentItem ? <spinner/> : ...}` showed the loading spinner whenever there was NO loaded item, regardless of whether `loading` was still `true` — so if the FIRST fetch (on mount) failed for any reason, the page showed a spinner forever (error box above it, spinner below it, contradicting each other) until some later action (GO) happened to succeed and finally push an item. Fixed: the spinner now only shows `!currentItem && loading`; once loading finishes with no item, a clear "failed to load, see error above" message shows instead. Also wrapped the whole mount effect in `try/finally` (it had no error handling at all — any thrown exception, e.g. a non-JSON response from `/api/folders/repos`, would leave `loading` stuck `true` forever with no error surfaced, silently).
3. **"combobox jest pusty caly czas"** — connects to the same root cause: if the mount effect's repos fetch failed, `repos` stayed `[]` with NO error shown (I never checked `reposRes.ok`). Fixed to surface an error. Separately: no fetch to `${CONTENT_PROVIDER_API_URL}/invoke` had a timeout, so a slow/unreachable API could hang the whole mount sequence indefinitely rather than failing fast — added a 15s `AbortController` timeout to `cp-flow.ts`'s `invokeCp`, matching what `cp-net-adapter` (the separate, earlier TS-rewrite session) already does for the same reason.
4. **The `"unexpected shape": {"success":true,"result":""}` error** — a REAL, distinct .NET behavior, not a bug in the sense of "wrong code": `GetItem` on a `loca` with no `config.yaml` (doesn't exist, not corrupted) returns an EMPTY response body, which `invokeCp`'s own fallback wraps as `{success: true, result: ""}`. The user's example (`8b603669-.../01` — Kamil's repo, loca "01") is exactly this: that loca genuinely doesn't exist under Kamil's repo (matches `chad-user-data-isolation.md`'s own note that Kamil's repo only has content under folder `02`+). Fixed `getItemByLoca` to detect this specific shape and throw a clean "Item not found" message instead of dumping the raw shape.
5. **"gdzies zostal usuniety dev panel"** — investigated (`git log` on `components/dev-panel/*`, `lib/flags.ts`): NOT touched by this Story or any of its commits (last real changes predate this Story entirely). `DEV_PANEL_ENABLED` (`lib/flags.ts`) defaults ON only when `NODE_ENV !== "production"` (plain `next dev`); a Docker build always sets `NODE_ENV=production`, which turns it OFF unless the build explicitly passes `ENABLE_DEV_PANEL=true` (a `Dockerfile` build arg that already existed before this Story). If the user was testing via a Docker-built image rather than `next dev`, the Dev Panel being hidden is the existing, designed-on-purpose behavior, not a regression from this Story's changes — reported back to the user as a finding, not silently "fixed" (there was nothing to fix in code that I touched).
