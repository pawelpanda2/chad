# Story 57 — Knowledge

Pointers used to do this specific Story, with why each was needed.

- `packages/net-content-provider/front_blazor/BlazorApp/Pages/Repos.razor` — the source of truth for the nav panel shape: repo selector, loca input, back/forward/GO/logout toolbar (confirmed live in this Story: the ←/↶/→ buttons all bind to the same `OnBackArrowBtnClicked` handler — dead/duplicate code, not a real "undo"/"forward" feature, so not worth replicating a second time).
- `packages/net-content-provider/front_blazor/BlazorApp/Components/ItemModels/TextView.razor` / `FolderView.razor` — per-item panels below the nav. Confirms the Folder `Body` is a `{index: name}` JSON map (same shape `cp-files`/`cp-net-adapter` already produce server-side) and that both panels are always mounted, each self-checking `Item.Type`.
- `packages/content-provider/entry` (`cp-entry`), `packages/content-provider/core` (`cp-core`) — the actual data-access layer this Story wires the dashboard up to. Built and verified against the real .NET Content Provider API in an earlier, separate session (not this Story) — this Story is the first real consumer of that work from inside the Next.js app.
- `documentation/dashboard/common/features/chad-user-data-isolation.md` — the reason "repo" is a read-only label here, not a picker: every other Content-Provider-touching endpoint in this dashboard is scoped strictly to the authenticated user's own `repoGuid`, resolved server-side via `getCurrentUserFromCookies()` and never accepted from the client. `cp-flow.ts`'s explicit-`repoGuid`-parameter pattern (as opposed to `dba`'s `AsyncLocalStorage`-based `runWithRepoContext`) is the one this Story follows, since `cp-entry`'s methods already take `repoGuid` explicitly — no new context-passing mechanism needed.
- `documentation/dashboard/common/features/responsive-layout-standard.md` + `packages/dashboard/components/shared/dashboard-page-shell.tsx` — the shared page-shell standard every dashboard page must use (`DashboardPageShell`, `toolbar` prop, `NavGroup` auto-appended). Note: `NavGroup`'s own Prev/Back/Forward is DASHBOARD-PAGE-level history (`useDashboardHistory()`), a different concept from this Story's Content-Provider-ITEM-level back/forward — the two coexist in the same toolbar row without conflict (this Story's Wstecz/Naprzód buttons are rendered via the `toolbar` prop, `NavGroup` is auto-appended after them by the shell itself).
- `packages/dashboard/app/api/folders/route.ts` (previous version) + `packages/dashboard/app/api/flow/cp-flow.ts`'s `getCurrentUserForms` — the code being replaced. That route/page pairing was a narrow, hardcoded actions/leads debug viewer, not a generic Content Provider browser; `getCurrentUserForms` is left in place (unused after this Story) rather than deleted speculatively.
- `packages/dashboard/next.config.ts` + `dba`'s existing `transpilePackages`/turbopack `resolveAlias` entries — the established pattern for pulling a local workspace package into the Next.js build, mirrored here for `cp-core`/`cp-net-adapter`/`cp-files`/`cp-entry`.
- `packages/dashboard/components/shared/error-box.tsx` — reused as-is for the error state, instead of writing a new one.
