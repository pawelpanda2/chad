# Bugs - run_dev.sh

## Bug: .env source error: Dashboard: command not found

**Severity:** HIGH

**Objawy:**
- `run_dev.sh` przerywa na ładowaniu `.env.02_local_mac`
- Log: `.env.02_local_mac: line 28: Dashboard: command not found`

**Przyczyna:**
- `.env.02_local_mac` był source'owany przez shell
- Linia 28 miała wartość `NEXT_PUBLIC_APP_NAME=Personal Dashboard`
- Wartość `Personal Dashboard` zawiera spację bez cudzysłowu
- Shell potraktował słowo `Dashboard` jako osobną komendę

**Poprawka:**
- Wartości ze spacjami muszą być w cudzysłowie: `NEXT_PUBLIC_APP_NAME="Personal Dashboard"`
- Komentarze w env muszą zaczynać się od `#`
- Zmienne muszą być w formacie `KEY=value` (bez spacji wokół `=`)
- Env source files muszą być zgodne jednocześnie z shell source i Docker Compose env formatem

**Czego NIE robić:**
- Nie dodawać opisowych nagłówków bez `#` (pierwsze linie pliku)
- Nie dopisywać markdownu do env
- Nie wypisywać sekretów z env w logach

**Naprawione pliki:**
- `.env.02_local_mac` - dodano cudzysłów do `NEXT_PUBLIC_APP_NAME`
- `.env.04_docker_mac` - dodano `#` do komentarzy i cudzysłów do wartości
- `.env.05_docker_qnap_test` - dodano `#` do komentarzy i cudzysłów do wartości
- `.env.06_docker_qnap_prod` - dodano `#` do komentarzy i cudzysłów do wartości
- `.env` - naprawiony (był uszkodzony, zawierał śmieci)

**Data naprawy:** 2026-06-10

## Bug: run_dev.sh uses Docker in local Mac flow

**Severity:** CRITICAL

**Objawy:**
- `run_dev.sh` z folderu `02_local_mac` sugeruje albo próbuje używać Dockera
- Pojawiają się komendy typu:
  `docker-compose -f docker-compose.mac.yml up content-provider-api`
- Lokalny dev flow miesza się z dockerowym flow
- Content Provider API nadal nie działa lokalnie jako zwykły proces
- Login przez GetByNames dalej pada, bo lokalny API process nie został uruchomiony

**Przyczyna:**
- Pomieszano odpowiedzialności folderów:
  - `02_local_mac` ma być BEZ Dockera
  - `04_docker_mac` jest od Dockera na Macu
- Próbowano naprawić lokalny dev przez docker-compose zamiast przez lokalne uruchomienie Content Provider API

**Poprawne zachowanie:**
- `02_local_mac/run_dev.sh` uruchamia Content Provider API jako lokalny proces `dotnet`
- `02_local_mac/run_dev.sh` uruchamia Personal Dashboard / Next.js jako lokalny proces
- Skrypt NIE używa Dockera, docker-compose, ani Dockerfile
- Skrypt robi healthcheck `/health` przed startem Next.js
- Jeśli CP API nie wystartuje, skrypt przerywa i pokazuje log

**Architektura folderów:**
```
03_scripts/nodejs/
├── 02_local_mac/          ← Local dev (NO Docker)
│   └── run_dev.sh         ← This script
├── 04_docker_mac/         ← Docker on Mac
├── 05_docker_qnap_test/   ← Docker on QNAP test
├── 06_docker_qnap_prod/   ← Docker on QNAP prod
└── 07_ssh_qnap/           ← SSH to QNAP
```

**Czego NIE robić:**
- Nie używać Dockera w `02_local_mac/run_dev.sh`
- Nie używać docker-compose w `02_local_mac/run_dev.sh`
- Nie odsyłać użytkownika do `docker-compose.mac.yml`
- Nie maskować problemu fallbackiem do SQLite
- Nie kontynuować Next.js, jeśli lokalny Content Provider API nie działa

**Komendy diagnostyczne:**
```bash
# Sprawdź healthcheck ręcznie
curl -i http://localhost:12024/health

# Uruchom CP API ręcznie (lokalnie, bez Dockera)
cd content-provider/charp/SharpContainerApi && dotnet run

# Zobacz logi
tail -f .tmp/content-provider-api.local.log
```

**Zmienione pliki:**
- `03_scripts/nodejs/02_local_mac/run_dev.sh` - usunięto Docker, dodano lokalny `dotnet run`, przygotowanie `.env`
- `app/api/auth/login/route.ts` - zwraca jasny błąd 503 gdy CP API niedostępne

**Data naprawy:** 2026-06-10

## Bug: Content Provider API local start uses dotnet run --no-build and stale .env

**Severity:** CRITICAL

**Objawy:**
- `run_dev.sh` próbuje uruchomić CP API
- healthcheck nigdy nie przechodzi
- log pokazuje:
  `No such file or directory ... bin/Debug/net8.0/SharpContainerApi`
- albo:
  `failed to read .env: line 1: unexpected character "#"`
- Next.js nie powinien startować, bo CP API nie jest healthy

**Przyczyna:**
- użyto `dotnet run --no-build`, mimo że lokalna binarka nie istnieje
- lokalny `.env` był stary / popsuty / qnapowy
- local dev nie przygotował `.env` z `.env.02_local_mac`

**Poprawka:**
- użyć `dotnet run` bez `--no-build`
- przed startem lokalnego CP API przygotować `.env` przez skopiowanie `.env.02_local_mac`
- lokalny flow nie może używać QNAP env
- dodać debug info: command, working directory, csproj path

**Czego NIE robić:**
- nie używać Dockera w `02_local_mac/run_dev.sh`
- nie używać `--no-build` w lokalnym dev, jeśli nie zbudowano wcześniej projektu
- nie kontynuować do Next.js, jeśli CP API nie jest healthy
- nie korzystać lokalnie z `.env.05_docker_qnap_test`

**Data naprawy:** 2026-06-10
