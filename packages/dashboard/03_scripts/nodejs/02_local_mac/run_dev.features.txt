# Features - run_dev.sh

## Local Development WITHOUT Docker

### Overview
`run_dev.sh` służy do **wyłącznie lokalnego** uruchomienia projektu na Macu, **BEZ UŻYCIA DOCKERA**.

### Architecture Separation

```
03_scripts/nodejs/
├── 01_envs/             ← Environment management
│   └── copy-env.sh      ← Sets active .env from named profile
├── 02_local_mac/        ← Local dev (no Docker)
│   └── run_dev.sh       ← This script
├── 04_docker_mac/       ← Docker on Mac
├── 05_docker_qnap_test/ ← Docker on QNAP test
├── 06_docker_qnap_prod/ ← Docker on QNAP prod
└── 07_ssh_qnap/         ← SSH to QNAP
```

### Feature: Environment Setup via copy-env.sh

**`run_dev.sh` always copies `.env.02_local_mac` to `.env` via `copy-env.sh` before starting local services.**

- `run_dev.sh` jest skryptem do zwykłego lokalnego uruchomienia na Macu
- Pierwszym krokiem skryptu ZAWSZE ma być ustawienie aktywnego env przez:
  ```bash
  bash 03_scripts/nodejs/01_envs/copy-env.sh local_mac
  ```
- Nie używać ręcznego `cp`
- Nie source'ować bezpośrednio `.env.02_local_mac`
- Dopiero po `copy-env.sh local_mac` skrypt ma czytać:
  ```bash
  source .env
  ```
- Content Provider API ma czytać ten sam `.env`
- Next.js ma czytać ten sam `.env`
- Dzięki temu local dev nigdy przypadkiem nie używa `.env` z QNAP test/prod

### Features

1. **Automatic Environment Setup**
   - Uses `copy-env.sh local_mac` to set active `.env`
   - Ensures consistent environment for all services
   - Prevents accidental use of QNAP env files

2. **Automatic Content Provider API startup (local process)**
   - Checks if CP API is already running via healthcheck
   - If not running, starts it as a local `dotnet run` process
   - Waits for healthcheck to pass before starting Next.js
   - Does NOT use Docker or docker-compose

3. **Healthcheck verification**
   - Uses `curl` to check `/health` endpoint
   - Maximum wait time: 60 seconds (30 attempts × 2 seconds)
   - Fails fast if CP API cannot be started

4. **Process management**
   - Tracks if CP API was started by this script
   - Cleanup on Ctrl+C stops CP API only if we started it
   - Logs saved to `.tmp/content-provider-api.local.log`
   - PID saved to `.tmp/content-provider-api.local.pid`

5. **Clear error messages**
   - If CP API fails to start, shows last 80 lines of log
   - Provides manual start command for debugging
   - Checks for dotnet SDK availability

### Flow

```
run_dev.sh
  ├─ copy-env.sh local_mac    ← Set active .env
  ├─ source .env              ← Load environment
  ├─ Check CP API health
  │   ├─ [OK] → Continue
  │   └─ [FAIL] → Check dotnet SDK
  │               → Start local dotnet run
  │               → Wait for health (max 60s)
  │               ├─ [OK] → Continue
  │               └─ [FAIL] → Show logs, exit 1
  ├─ Install npm dependencies (if needed)
  └─ Start Next.js dev server
```

### Files

- PID file: `.tmp/content-provider-api.local.pid`
- Log file: `.tmp/content-provider-api.local.log`

### Manual commands

```bash
# Set local env manually
bash 03_scripts/nodejs/01_envs/copy-env.sh local_mac

# Check health manually
curl -i http://localhost:12024/health

# Start CP API manually (local)
cd content-provider/charp/SharpContainerApi && dotnet run

# View logs
tail -f .tmp/content-provider-api.local.log
```

### Notes

- This script does NOT use Docker
- Docker-based development is in `04_docker_mac/`
- TypeScript Content Provider (`content-provider/typescript`) is NOT active
- Local dev uses C# API via `dotnet run`
- CP API listens on port 12024 (localhost)
- Next.js dev server listens on port 12020 (localhost)
- Next.js starts ONLY after CP API is healthy
