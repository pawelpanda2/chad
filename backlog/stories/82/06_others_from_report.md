## cp-gui duplication (confirmed, not resolved)

Confirmed real duplication: `packages/cp-gui`'s `ContentProviderBrowser`/`FolderView`/`TextView` and Dashboard's `folders/page.tsx` are two independent ports of the same Blazor source. cp-gui's own `BackendAdapter` talks to `cp-api` (the .NET Content Provider's HTTP API), which no longer has real writes (`put`/`postParentItem` still throw, "Stage 3") and isn't deployed at all in this stack anymore (Story 72). Proposal for later: either (a) retire `cp-gui`'s dashboard-facing ambition and keep it purely as the future standalone CP browser once cp-api itself grows a `dba`-backed write layer, or (b) extract Dashboard's now-working write logic into shared, framework-agnostic components `cp-gui` can re-export and Dashboard can import directly (skipping cp-api entirely for the in-dashboard case). Not attempted here ‚ÄĒ real architecture decision, out of scope for a bug-fix Story per the task's own instruction.
