# Story 57 — Others from report

## Architectural decisions (made autonomously, no user available to ask)

- **Full replacement, not addition.** The prior `/dashboard/folders` page (a narrow, hardcoded actions/leads debug viewer with a "Developer logs" panel) and its `/api/folders` route were replaced outright, not kept alongside the new generic browser. `getCurrentUserForms` (in `app/api/flow/cp-flow.ts`) is now unused but was left in place rather than deleted speculatively — it's a shared module function and this Story didn't audit every other possible caller.
- **New `cp-gui` components were NOT reused directly.** An earlier, separate session built a standalone `packages/cp-gui` component library intended partly for exactly this dashboard-embedding use case, but its components carry a free repo picker and `cp-plugin` (local desktop) integration that don't fit this dashboard's per-user isolation model or its web-only deployment. This Story wrote new, dashboard-native components instead.
- **`cp-entry`/`cp-core` were tried first, then dropped entirely** (see `02_plan.md`'s "Correction" section for the full story). First attempt reused the earlier session's TypeScript Content Provider rewrite as the data-access layer; a real deploy attempt failed because the dashboard's Docker build never builds those packages' `dist/` output, and the follow-up instruction was to not depend on them at all — this Story's actual data-access layer ended up being a small addition (`getItemByLoca`) to the dashboard's own existing `cp-flow.ts`, the same direct-to-.NET-API mechanism every other dashboard page already uses. **Lesson for future Stories that add a new workspace dependency to `packages/dashboard`:** check `packages/dashboard/Dockerfile` for a matching `RUN pnpm --filter <pkg> build` step (mirroring the existing `dba` one) BEFORE the first deploy, not after a failed one — or better, per this Story's actual resolution, prefer extending `cp-flow.ts` over adding a new workspace dependency at all when the need is just "call the existing .NET Content Provider," since that's already a proven, zero-extra-Docker-config path.
- ~~**Repo is a read-only label, not a picker**~~ — **reverted, see below.** First-pass decision, made necessary (at the time) by this dashboard's existing per-user data isolation model. The user corrected this directly (Input 3/4): the reference screenshots show a real dropdown over all 36 repos, from Pawel's own use of the standalone Blazor tool. Resolved not by abandoning the isolation model, but by narrowing the exception to exactly one login (`pawel_f`) — see the new decision below.
- **Back/Forward is a real, working history stack**, not a port of Blazor's own dead-code triple-button (all three of `←`/`↶`/`→` call the identical handler in the real Blazor source — confirmed by re-reading `Repos.razor` in this Story, not assumed from memory). Kept even after the corrections, since it's a confirmed dead-code artifact, not a real feature to replicate.
- **Full repo picker gated to `pawel_f` by username, not a role/admin flag** (Correction 2) — this codebase's user model (`lib/user-service.ts`, `lib/session.ts`) has no admin/role concept at all. Username-gating is a narrower, more fragile mechanism than a real role check (it breaks silently if the username ever changes, and doesn't generalize to a future second trusted operator) — done this way because it was the only option available without inventing new user-model infrastructure as a side effect of this Story. If more than one person should ever get full repo access, this needs a real role field, not another hardcoded username.
- **Per-item buttons rendered but disabled, not omitted** (Correction 2, reversing the first pass) — Folder/Content/Config/Terminal (`cp-plugin`), GoogleDoc/Tts (no integration exists), and Add/body-Save (writes) are all visually present, matching the screenshots, but non-functional with an explanatory tooltip. This is a deliberate middle ground: visual fidelity without silently taking on either a `cp-plugin` bridge or a write-access-control decision that wasn't asked for.

## Limitations / what was deliberately left undone

- **No actual browser click-through verification.** This environment has no headless-browser or screenshot tool. Verification was done via: TypeScript build (`tsc --noEmit`, clean), a real `next dev` server, real authenticated sessions (`pawel_f`/`changeme`), `curl` calls against the real, already-running .NET Content Provider API confirming correct data end-to-end (including two full rounds cross-checked against the user's own pasted reference screenshots and raw JSON), and a real, complete `docker compose build dashboard` (not just a local approximation) after the Docker-related correction. React state logic (history stack, disabled-button conditions) verified by code review only.
- **No write operations.** A real write path exists in `cp-flow.ts` (`Put`, already used by other flows) and could be wired to the Add/Edytor-Save controls this Story added but left disabled — deliberately not done, because doing so now (combined with the just-added free repo picker) would mean deciding "which repos can `pawel_f` WRITE to via this browser" without being asked to make that call.
- **`403`-for-non-owner repoGuid path** (in `/api/folders`) verified by code review and by the identical, already-tested condition in `/api/folders/repos`, but not exercised against a second real non-`pawel_f` login end-to-end (no second test password available this session).
- ~~**The "combobox still empty" report (Input 5/6) was never reproduced in this environment.**~~ — **RESOLVED in Correction 5 (Input 7, Task 10).** Root cause: the repo-root `.dockerignore` had a `**/repos/` glob that matched *any* directory named `repos`, including `packages/dashboard/app/api/folders/repos/` — Docker silently dropped that whole API route from every build image since the route was created. This is why every prior `curl` test looked fine (they ran against `next dev`, which isn't subject to `.dockerignore` at all) while the real Docker deployment's combobox stayed empty. Fixed by removing that one line; confirmed by rebuilding the real image and re-testing the exact failing request end-to-end. **Lesson for future Stories:** a passing `next dev` test and a passing `tsc`/`next build` inside the image are NOT sufficient to confirm a new API route actually ships — `.dockerignore` (and the Docker build context generally) can silently drop files with no build error at all, since a missing route directory just compiles to "one fewer route," not a failure. When a Docker-deployed symptom can't be reproduced via `next dev`, `docker exec <container> find /app -path "*<route>*"` against the actually-running image is a fast way to tell "route not shipped" apart from "route shipped but buggy."
