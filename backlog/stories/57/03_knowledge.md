# Story 57 — Knowledge

Pointers used to do this specific Story, with why each was needed.

- `packages/net-content-provider/front_blazor/BlazorApp/Pages/Repos.razor` — the source of truth for the nav panel shape: repo selector, loca input, back/forward/GO/logout toolbar (confirmed live in this Story: the ←/↶/→ buttons all bind to the same `OnBackArrowBtnClicked` handler — dead/duplicate code, not a real "undo"/"forward" feature, so not worth replicating a second time).
- `packages/net-content-provider/front_blazor/BlazorApp/Components/ItemModels/TextView.razor` / `FolderView.razor` — per-item panels below the nav. Confirms the Folder `Body` is a `{index: name}` JSON map (same shape `cp-files`/`cp-net-adapter` already produce server-side) and that both panels are always mounted, each self-checking `Item.Type`.
- ~~`packages/content-provider/entry` (`cp-entry`), `packages/content-provider/core` (`cp-core`)~~ — **superseded, see `02_plan.md`'s "Correction" section.** First attempt wired the dashboard up to these (an earlier, separate session's TypeScript rewrite of the .NET Content Provider); reverted after a failed deploy showed the Dockerfile never builds these packages, and a follow-up instruction said not to add them at all — this Story now goes through `cp-flow.ts` instead, the same path every other dashboard endpoint already uses.
- `packages/dashboard/app/api/flow/cp-flow.ts` — the actual data-access layer this Story ends up using: its own header comment states it's "the ONLY place where Content Provider args are manually constructed." Its low-level `invokeCp()` (`fetch` to `${CONTENT_PROVIDER_API_URL}/invoke`) is the same mechanism every other dashboard endpoint (forms, leads, statuses, etc.) already relies on in production — this Story adds one new exported function, `getItemByLoca`, calling `invokeCp` directly rather than the higher-level `invokeCpWithTrace` (which only extracts a `.result` string, shaped for the Body-only forms/leads flows, not `GetItem`'s real `{Body, Settings}` response).
- `documentation/dashboard/common/features/chad-user-data-isolation.md` — establishes the per-user isolation model every other Content-Provider-touching endpoint follows; `lib/user-service.ts`/`lib/session.ts` were checked (Correction 2) and confirmed to have NO admin/role flag, which is why the eventual repo-picker exception (`pawel_f`-only) is gated by username rather than a proper role check — a narrower, more fragile mechanism, flagged as such rather than presented as a clean solution. `cp-flow.ts`'s explicit-`repoGuid`-parameter pattern (as opposed to `dba`'s `AsyncLocalStorage`-based `runWithRepoContext`) is the one this Story follows — no new context-passing mechanism needed.
- `packages/net-content-provider/front_blazor/BackendAdapters/Workers/RepoAdapter.cs` + `Names/SNames.cs` (Correction 2) — the real args for listing all repos: `["IRepoService", "IMethodWorker", "GetAllReposNames"]`. Verified live directly against the real .NET API (`curl .../invoke`) before writing any code, confirming 36 real repos and the exact `[{Body: null, Settings: {id, name, type, address}}, ...]` response shape.
- `documentation/dashboard/common/features/responsive-layout-standard.md` + `packages/dashboard/components/shared/dashboard-page-shell.tsx` — the shared page-shell standard every dashboard page must use (`DashboardPageShell`, `toolbar` prop, `NavGroup` auto-appended). Note: `NavGroup`'s own Prev/Back/Forward is DASHBOARD-PAGE-level history (`useDashboardHistory()`), a different concept from this Story's Content-Provider-ITEM-level back/forward — the two coexist in the same toolbar row without conflict (this Story's Wstecz/Naprzód buttons are rendered via the `toolbar` prop, `NavGroup` is auto-appended after them by the shell itself).
- `packages/dashboard/app/api/folders/route.ts` (previous version) + `cp-flow.ts`'s `getCurrentUserForms` — the code being replaced. That route/page pairing was a narrow, hardcoded actions/leads debug viewer, not a generic Content Provider browser; `getCurrentUserForms` is left in place (unused after this Story) rather than deleted speculatively.
- `packages/dashboard/Dockerfile` — read after the first deploy attempt failed; confirmed it only builds `dba`'s `dist/` output before `next build`, nothing else — the direct cause of the `Module not found: 'cp-entry'`/`'cp-core'` failure. Resolved by removing the dependency rather than extending the Dockerfile (see `02_plan.md`).
- `packages/dashboard/components/shared/error-box.tsx` — reused as-is for the error state, instead of writing a new one.
